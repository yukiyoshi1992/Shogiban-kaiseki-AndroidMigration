## ⚠️ 最優先・セッション開始時、調査ではなくこの手順をそのまま実行すること（2026-06-21確定）

このプロジェクトはユーザーとDiscord経由でも連絡を取り合う運用（`01 企画・管理/handover.md`参照）。Discordツール（`reply`/`fetch_messages`等、`plugin:discord:discord`提供）が読み込まれない事象が繰り返し発生し、ユーザーの時間を大きく無駄にした。**原因は確定済み・対処法も確定済み**。次回以降のセッションは原因をもう一度調査し直す必要はない。以下を**毎セッション開始時、本格的な作業に入る前に必ず実行**すること。

**実行手順（2026-06-23訂正版。調査不要、そのまま実行）：**
1. ToolSearchで`select:mcp__plugin_discord_discord__reply`等を確認し、Discordツールが読み込まれているか見る。
2. **読み込まれているかどうかに関わらず**、毎回必ずBashで次の2コマンドを実行する（`dangerouslyDisableSandbox: true`を付ける。プロジェクト外のホームディレクトリ配下のため）。**重要：対象ディレクトリは`marketplaces/.../external_plugins/discord`ではない**——実際にMCP起動時に使われる`CLAUDE_PLUGIN_ROOT`は`~/.claude/plugins/cache/claude-plugins-official/discord/<バージョン番号>/`（`.in_use/`マーカーファイルの生成タイミングで確認済み）。バージョン番号は変わる可能性があるため`*`でglobすること：
   ```
   cd ~/.claude/plugins/cache/claude-plugins-official/discord/*/ && bun install --no-summary
   bun -e "await import('discord.js'); await import('@modelcontextprotocol/sdk/server/index.js');"
   ```
   1行目（`bun install`）は依存解決の確認用（通常0.2秒、ほぼ常に瞬時）。**2行目が本当に重要な手順**：discord.js等の依存モジュールを実際にインポートし、ファイルへの「初回アクセス」を今のうちに済ませておく（詳細は下記2026-06-23朝の追加調査を参照、初回13.66秒・2回目以降0.64秒という実測差を確認済み）。これを毎セッション開始時に行うことで、Claude Code本体がMCPサーバーを起動する際にはこの初回アクセス分の遅延を払わずに済む。
3. 手順1でツールが見つかっていれば、それで完了。ユーザーに繋がっている旨を一報して本来の作業に進む。
4. 手順1でツールが見つからなければ：手順2を実行済みであることを踏まえ、**ユーザーに「このターミナルセッションを終了して新しいセッションを開始してください」と伝える**（`/reload-plugins`では直らない、既知の制約）。新セッションでまた手順1〜2を実行する。

**この問題の背景（参考情報、対処には不要）**：Discordプラグインのサーバーは起動時に毎回`bun install --no-summary && bun server.ts`を実行する設計（`package.json`の`start`スクリプト）。**Discord側（トークン有効性・Gateway到達性・Intents設定）はこの問題と無関係で、無罪と確認済み**（discord.jsの`Client.login()`単体テストで数秒で`ready`に到達する）。

## What this project is

A native Android app + small FastAPI server that lets a user record a shogi (将棋) game as a KIF file by photographing the board after every move, with the board-recognition CNN doing the actual reading. This is the **successor** to a finished predecessor project, `\\YukiYoshiNAS\Shogiban-kaiseki-tool` (a Streamlit/Python desktop tool that proved the recognition pipeline works end-to-end). That predecessor is **done and archived as of 2026-06-20** — do not edit it; treat it as a read-only reference and a source of reusable code/model/lessons-learned. This repo starts at the requirements-definition stage; nothing has been built here yet (`03 設計・開発/01 Androidアプリ開発/` and `02 API開発/` are empty).

For session-to-session continuity and current status, see `01 企画・管理/handover.md` — read it alongside this file before doing anything.

**New sub-project started 2026-06-23, separate continuity docs**: `2026年6月24日 CNN処理オンデバイス化/` is a verification-only experiment (does running the board-recognition CNN on the Android device instead of the PC server improve response time?) — explicitly does **not** touch this main app's existing source (server or Android), develops in a duplicated environment instead, and only migrates to production if the on-device approach is clearly better. It has its own `01 企画・全体管理/handover.md` and `02 要件定義/要件定義.md` — read those (not this file) when working on that sub-project.

## Why this project exists (carried over from `01 企画・管理/課題.txt`)

The predecessor's actual operational pain points, which this rebuild is meant to solve:
1. **No real-time-ness.** The old flow was: phone camera app → folder-sync app → NAS folder → Python watches the folder. Sync only runs every few minutes, so there's dead time after taking a photo before the system reacts, and the operator can't tell when it's safe to take the next photo.
2. **Too many apps/devices in the loop.** The phone needs a separate camera app, a separate sync app, and a separate file-manager app; when automatic detection fails, a human has to go to the *server* to do manual calibration. Sharing/handing off the finished KIF (e.g. to KENTO for game analysis) isn't integrated either.
3. **Orientation isn't fixed.** The old camera app lets photos come in portrait or landscape unpredictably, so the pipeline has to *detect* orientation (blue-triangle marker) and has a whole human-fallback path for when that detection is wrong. The plan for this project is to control the camera (build it into the native app) so orientation is fixed by construction and this detection step disappears entirely.

Resolution direction already decided (`02 要件定義/ネイティブアプリ化_要件.xlsx`): a native Android app owns the camera and the whole user flow; it talks to a small FastAPI server (on the same Windows PC that used to run the Streamlit tool) over HTTP instead of folder-sync, for real-time turnaround; the recognition logic itself is reused from the predecessor project, not rewritten.

- **Calibration UX overhaul, driven by real-device testing (2026-06-21)**: first real on-device test of the calibration screen surfaced three usability problems, all now fixed:
  1. **Camera preview looked more zoomed-in than the actual capture** (board didn't appear to fit, but the captured photo actually did) — root cause: `PreviewView`'s default `scaleType` (`FILL_CENTER`) crops the preview to fill the view. Fixed by setting `PreviewView.ScaleType.FIT_CENTER` in both `CalibrationScreen`'s `CameraCaptureStep` and `PlayScreen` — user confirmed the actual captured photo already included the full board even before this fix, so this was a display-only bug.
  2. **Manual 4-corner tapping was "too hard, always off"** — fixed two ways together: (a) **red-circle auto-calibration now runs first** (see corrected note above) — `POST /calibration/photo` now accepts `points` as optional; omitting it triggers `recognition.calibrate_from_image()` (ported red-circle detection), returning `{"status": "calibration_failed", "reason": "red_circles_not_found"}` if no circles are found, at which point the Android client falls back to manual tapping on the same already-captured photo (no retake needed). Also widened from `KEYCODE_VOLUME_UP`-only to **both volume up and down** per explicit user request.

## Hard-won lessons to not relearn the expensive way

- **Orientation bugs disguise themselves as model/calibration weaknesses, and Claude cannot detect them by reasoning about pixels.** A 180°-misrotated shogi board still looks "locally plausible" (the board is physically point-symmetric, so a flipped board produces a coherent-looking but wrong recognition pattern, not obvious garbage). The predecessor project burned a long multi-session detour on this (see its `CLAUDE.md` "001-raw ... RESOLVED" section) before discovering the real bug was a 180° orientation error, not a CNN weakness. **Whenever orientation is in play (which, per this project's whole purpose, should become rare — but if it ever resurfaces, e.g. in calibration debugging), get a human to look at the actual image and confirm sente-left before trusting anything else.** "Two independently-computed calibrations agreeing with each other" is *not* proof orientation is correct — both can independently encode the same wrong orientation.
- **Don't trust Claude's own visual reading of board photos.** Present model output (labels/coordinates/confidence) and ask the human to verify against the photo; don't have Claude claim "I can see piece X moved to Y" from an image.
- **A human nudging/straightening a physical piece mid-game, with no real move occurring, is a real production scenario**, not a hypothetical — it caused a long failure cascade in the predecessor before `classify_frame`'s board-anchored (rather than photo-to-photo) redesign fixed it. Any reimplementation must keep comparing against the authoritative tracked board state, never the previous photo.
- **Silent mislabeling/zero-warning failure modes are common in this domain** — e.g. a wrong row count in a labeling pattern shifted every label by one row with no warning at all (predecessor `CLAUDE.md`, `label_placement.py` section). If this project ever needs to extend training data, budget time for a manual visual row/col-overlay sanity check, not just trusting tool output.
- **Console output on this Windows/network-share setup is sometimes cp932-encoded** and mangles Japanese text when redirected/captured (e.g. through a background task buffer) — re-decode, don't assume the underlying data is actually wrong. Set `PYTHONIOENCODING=utf-8` on one-off Python commands that print Japanese text.
- **Custom UI components (e.g. a corner-tap calibration UI) can look "completely broken" to a non-technical tester for boring reasons** (clicking the wrong element, browser caching) — when a UAT report says "X doesn't work at all," reproduce it yourself (headlessly if possible) before assuming a code regression. (This came up in the predecessor's Streamlit corner-click UI; will likely recur with this project's native 12-point tap calibration UI.)
- **`@old/` convention**: this user prefers moving superseded files into an `@old/` subfolder at the same directory level rather than deleting them. Carry this forward if it comes up.
- **Git on this network share** may report "detected dubious ownership". Never run `git config --global --add safe.directory` from an agent session (changes global config) — use the one-off `git -c safe.directory='*' <command>` prefix instead, or ask the user to fix it globally themselves.
- **Image I/O on Windows with Japanese paths**: use `cv2.imdecode(np.fromfile(path, dtype=np.uint8), ...)` / `cv2.imencode(...)[1].tofile(path)`, not `cv2.imread`/`cv2.imwrite` (the latter silently fail on non-ASCII paths).
- **`detect_red_circles` (auto corner detection) can false-positive on any small saturated-red object in the background, not just the 4 calibration stickers.** Confirmed 2026-06-23: a laptop's red "AMD Radeon" logo sticker sitting in frame got picked as one of the 4 corners, producing a wildly wrong perspective transform (the grid-confirm overlay rendered as a single flat maroon color — diagnosed via `recognition.detect_red_circles`/`select_board_corners` on the saved `runtime/photos/*.jpg`, not by guessing from the screenshot). This is environmental, not a code bug — when this happens, either keep red objects out of frame, or use the existing "手動タップで直す" fallback (this is exactly the scenario the 5回目UAT課題④ human-confirmation step was built for).

**Fix**: `AndroidManifest.xml`'s `<activity android:name=".MainActivity">` now declares `android:configChanges="keyboard|keyboardHidden|navigation"` — this tells Android "I'll handle these configuration changes myself" (in practice, doing nothing, since the app doesn't care about hardware keyboard/navigation availability), which suppresses the destroy-and-recreate entirely. The Activity now just receives a (currently unhandled, and not needed) `onConfigurationChanged()` callback instead.

## 8回目テスト課題⑩：赤丸キャリブレーションの精度低下（原因特定・修正済み、180枚の過去写真で回帰ゼロを確認）

1. 強い候補（`CIRC_MIN`=0.55以上）だけで4点以上ある場合のみ、その中から面積最大の4点コンボを直接採用（`_best_quad_by_area`）。届かない場合に円形度を緩めて4点直接採用には絶対に回さない（これが元の課題⑩の原因）。

## 8回目テスト課題⑪：Web版KIF一覧、HEADリクエストが405で弾かれていた（原因確定・修正済み）

7回目UAT課題⑨（Web版一覧の2回タップ問題）の対策として入れたHEAD probe＋リトライ方式（commit `0725535`）をユーザーが実機に反映したところ、**遷移自体が完全にできなくなった**との報告。サーバログに`HEAD /web/games/{id} HTTP/1.1" 405 Method Not Allowed`が複数回記録されていた。

`/web/games`・`/web/games/{game_id}`が`@app.get(...)`でGETのみ登録されており、`_WEB_NAV_SCRIPT`がクリック時に送るHEADリクエストが毎回確実に405で弾かれていたことを、`FastAPI`の`TestClient`で直接再現して確認（想定では自動でHEADも許可されるはずだったが、このバージョンでは違った）。これは7回目UAT課題⑨「2回タップしないと遷移しない」の真因だった可能性が高い——HEAD probeは毎回100%失敗するため、4回のリトライ（0.5秒間隔、計2秒)をすべて消化してから最終フォールバック（`location.href`への直接遷移）に落ちる動きになっていたはずで、LAN側の偶発的な詰まりではなく確実に再現するルーティングの不備だった。

**修正**：`@app.get(...)` → `@app.api_route(..., methods=["GET", "HEAD"])`に変更（`web_games_list`・`web_game_detail`の両方）。`TestClient`で`HEAD`が200（body 0バイト）になることを確認済み、`GET`の動作も変化なし。

**続報：HEAD 405修正だけでは直らなかった。** ユーザーが実機で確認したところ、HEADは200 OKになったが「ロードゲージが伸びきらず止まる」症状は継続。ユーザーから「過去の経緯にとらわれず、なぜスマホだけ失敗するのか調査してほしい」との明示の指示を受け、調査をリセット。

まず、HEAD→同一keep-alive接続で直後にGETを送るとサーバー側で何か壊れないか（フレーミング崩れ・ハング等）を生ソケット通信で直接実験——**再現せず**（正常に200を返し、ハングしない）。サーバー側の接続再利用バグという説は後退。

次に、画面上に診断ログ（タップ→probe結果→`location.href`呼び出し→3秒後も同じページか、を画面下部に表示）を追加し、ユーザーから実機のスクリーンショットを取得。これが決定的な証拠になった：

```
[0ms] tap: /web/games/20260623_14-27
[0ms] probe attempt 0 -> ...
[45ms] probe OK
[46ms] calling location.href now
[3034ms] STILL HERE 3s after location.href - navigation did not happen
```

**probeのfetch自体は23〜46msで毎回成功し、その直後に`location.href`代入も確実に呼ばれている（ログに記録される）にもかかわらず、3秒後も同じページに留まり続けていた。** これはLANの詰まりでもサーバー側の問題でもなく、**Promiseの`.then()`という非同期コールバック内から行う`location.href`代入が、このスマホのブラウザでは実質的に無視されていた**ことが直接の原因だった。クリックを横取りしてfetchを挟む方式そのものが、素の同期的なリンククリック（ブラウザネイティブの遷移）が持つはずの効果を失わせていたと考えられる——「LANが2回目の通信を詰まらせる」という当初からの前提自体が、実は当てはまっていなかった可能性が高い（HEAD probeは常に数十msで成功しており、LAN側の遅延は一度も観測されていない）。

**最終対応（2026-06-23、3回目）**：JSによるクリック横取り・fetch probe・`location.href`呼び出しを全廃し、`_WEB_NAV_SCRIPT`を空文字列にして`<a href>`のみのブラウザ標準遷移に戻した（6回目UAT③で確定済みの`:hover`スコープ修正＝`@media (hover: hover) and (pointer: fine)`はそのまま維持）。HEAD probeの実験でサーバー応答自体が速いことは確認済みのため、素のクリックで詰まる理由はない。`/web/games`・`/web/games/{game_id}`の`methods=["GET","HEAD"]`自体は無害なため変更していない（HEAD自体が悪さをしていたわけではなく、JSの非同期`location.href`が原因だったため）。

**続報：JS撤廃版でも「動かない」と報告されたが、実際は別の問題（ブラウザキャッシュ）だった。** ユーザーから再度スクリーンショットが届いたが、画面下部に**削除したはずの診断ログJSがまだ表示されていた**——これはJS撤廃が効かなかったのではなく、**スマホのブラウザが古いHTML（前のJS入りバージョン）をキャッシュして再表示していた**ことを意味する。`web_games_list`/`web_game_detail`がこれまで`Cache-Control`ヘッダーを一切返していなかったため、ブラウザの既定のヒューリスティックキャッシュに乗ってしまっていたと考えられる（ログにあった謎の`HEAD /web/games`連発・`/health`への到達も、古いJSが裏で動いていた跡と考えられる）。

**対応**：両ルートの返り値を`HTMLResponse(content=html, headers={"Cache-Control": "no-store, no-cache, must-revalidate"})`に変更（`_NO_STORE_HEADERS`定数）。`TestClient`で実際にヘッダーが返ることを確認済み（commit `d80e6b3`）。

**続報：キャッシュ修正で遷移自体は直ったが、「2回タップ」自体は残った。ただしユーザーの観察で決定的な手がかりを得た。** ユーザー報告：「1回目のボタンと2回目のボタン、違うものを押しても2回目は遷移する。1回目は絶対遷移しないが、2回目は同じボタンである必要がない」。サーバーログでも同じパターンを確認——**1回目のタップに対応するHTTPリクエストが一度もサーバーに記録されず**、2回目のタップで初めて1件のGETが記録される、という挙動が何度も繰り返された。

JSによるクリック横取りは現在完全に存在しない（素の`<a href>`のみ）ため、「クリックイベントは発生したが遷移処理でつまずく」という中間状態はもう存在しえない。つまり**1回目のタップはDOMのクリックイベントとしてそもそも発生していない**と判断できる。対象に依存しない（どのリンクをタップしても1回目は無反応）という特性は、「ページ読み込み直後の最初のタップが、ページの中身ではなくブラウザ自身（アドレスバーの縮小・タブのフォーカス確立等）に消費される」というAndroid系ブラウザで知られる現象と一致する。

**対応（実験、2026-06-23）**：`_WEB_NAV_SCRIPT`に、クリック横取りを一切行わない最小限のスクリプトを追加——`window.addEventListener('load', () => { window.scrollTo(0, 1); window.focus(); })`。ページ読み込み完了時にブラウザのUI状態を先に確定させ、ユーザーの最初のタップが「ブラウザの初期化待ち」に消費されることを防ぐ狙い。クリック・fetch・location.href操作は一切行わない（前回の教訓——非同期コールバックからのナビゲーション操作は問題を起こしうる——を踏まえ、素のリンク遷移そのものには触れない設計）。commit `9c224ba`。

**続報：scrollTo/focus緩和策も効果なし。さらにユーザーが別ブラウザのシークレットモードでも検証し、同症状を確認。** これによりブラウザ側の原因（キャッシュ・拡張機能・サイト設定）は完全に除外できた——**残るはOS/端末本体側の挙動**という結論に至った。

実機はOPPO（ユーザー確認済み）。ColorOSには「誤操作防止」「誤タッチ防止」「タッチ保護」のような名前で、画面切り替え直後の最初のタップを誤操作とみなして無視する機能が存在することがある（メーカー・バージョンにより名称が異なる）。ユーザーに設定アプリ内をそのキーワードで検索し、見つかればオフにして再検証するよう依頼した。

**現時点の整理**：Webページ側で考えられる対策（HEAD 405修正、ブラウザキャッシュ修正、JS制御全廃、scrollTo/focus緩和）はすべて試したが、いずれも症状の本質には効かなかった。OS側の設定が見つからない場合、これはWebページの実装では解決できない端末側の制約として受け入れる、という判断も視野に入れてユーザーに伝えた。

**真因確定（2026-06-23、解決）**：ユーザーがChrome DevTools（USB inspect）を接続し、Console上で決定的な実験を実施。
1. `touchstart`/`touchend`/`pointerdown`/`pointerup`/`mousedown`/`mouseup`/`click`/`focus`/`blur`を`document`に`capture:true`で登録するコードを実行 → **それ以降は1回のタップで遷移するようになった**。
2. 比較実験として、中身のないコード（`0`を入力するだけ）を実行 → **引き続き2回タップが必要だった**。

この2つの結果から、「Consoleで何か実行すること自体」や「DevToolsが繋がっていること自体」の副作用ではなく、**実際にタッチイベントのリスナーをdocumentに登録すること自体**が直接の修正効果を持っていたと確定した。これは、ページが（パッシブな）タッチリスナーを一つも持たない状態だと、一部のAndroid端末（このOPPO機含む）ではブラウザが「このページはタッチを使わないだろう」と判断し、読み込み直後の最初のタッチ入力の処理を最適化・後回しにすることがある、というモバイルWebで知られた挙動と一致する。touchstartリスナーを1つ登録するだけでこの最適化が働かなくなり、入力が即座に処理されるようになる。

**最終対応**：`_WEB_NAV_SCRIPT`を`document.addEventListener('touchstart', function () {}, { passive: true, capture: true });`のみに変更（クリック横取り・fetch・location.href操作は一切なし、本当にリスナー登録だけ）。`passive: true`はユーザーの実験で使われた挙動（明示的にpassiveを指定しなかった場合、Chromeはdocument/window上のtouchstart/touchmoveリスナーを既定でpassiveにする仕様）と一致させた。commit `0c43e32`。push済み、**ユーザーに最終確認依頼中**。

これでWeb版KIF一覧の「2回タップしないと遷移しない」問題（7回目UAT課題⑨〜8回目テスト課題⑨）は、HEAD 405・ブラウザキャッシュ・touchstartリスナー欠落という**3つの独立した原因**が積み重なっていたことが分かった。当初から有力候補だった「LANが2回目の通信を詰まらせる」という説は、結局どの段階の症状にも当てはまっていなかった（推測だけで終わらせず、実機での直接実験で1つずつ確定させたことが解決につながった）。
