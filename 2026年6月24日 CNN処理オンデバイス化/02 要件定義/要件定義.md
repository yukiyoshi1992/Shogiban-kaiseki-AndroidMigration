# CNN処理オンデバイス化 検証 要件定義

それ以外は実施したいこと.txtの前提・本プロジェクトの既存の方針から合理的に判断した内容。
確認後、このファイルを直接更新していく想定（前PJ・本PJの「要件定義に時間をかけすぎない、

## 1. 目的・背景（`01 企画・全体管理/実施したいこと.txt`より）

現状、CNNによる盤面認識は本番サーバー（PC、`00 runtime/server/`）側で行っており、
スマホは撮影した写真をサーバーへ送信している。この「写真送信」がレスポンス低下の
要因になっている可能性があり、**CNN処理をスマホ側（オンデバイス）で行った場合に
レスポンスが改善するかどうかを検証する**のが目的。

**前提・ルール（確定済み）**：
- 検証目的のみ。既存ソース（`00 runtime/server/`・既存Androidアプリ）は変更しない。
- 新しく複製した環境で新規開発し、既存（サーバー側CNN）と動作を比較する。
- 明らかにスマホ側CNNの方が良いと判断された場合のみ、本番移行を検討する（今回は検証のみで移行作業自体はスコープ外）。
- 作業はすべて `2026年6月24日 CNN処理オンデバイス化/` 配下（`フォルダ構成ルール.txt`準拠）。

## 2. 検証スコープ（要確認）

**推奨：CNN推論だけでなく、画像から9×9認識結果を得るまでの「認識パイプライン全体」をオンデバイス化する。**

理由：CNN推論（`predict_board`）だけをオンデバイス化しても、透視変換用の元画像（盤全体の高解像度写真）をサーバーに送る必要が残るなら、検証したい「写真送信がボトルネックかどうか」を切り分けられない。`predict_board`が受け取る入力（透視変換済みの900×900画像）を作る`warp_board`（OpenCVの`getPerspectiveTransform`/`warpPerspective`、4隅タップ座標から計算）まで含めてオンデバイス化し、**サーバーに送るのは9×9のラベル配列（数十バイト程度のJSON）だけ**にする。

**スコープ外とする（推奨）**：
- `classify_frame`（指し手判定・合法手探索、`python-shogi`依存）は今回は移植しない。9×9ラベル配列をサーバーに送り、判定自体は引き続きサーバー側の既存ロジックで行う。理由：このロジックは将棋ルールの正しさが重要で、Kotlin側への移植は別途バグの出やすい大きな作業になり、今回の検証目的（レスポンス比較）には不要。
- 赤丸自動検出（`detect_red_circles`/`select_board_corners`）も今回は対象外。4隅タップ（手動）の写真だけで比較する想定——自動検出の有無は本検証の結果に影響しない。
- KIF生成・対局状態管理（`session.py`）はそのまま既存サーバーを使う。

**要確認**：上記スコープ（CNN＋透視変換まで／指し手判定以降はサーバー）で問題ないか。

## 3. モデル変換の実現性調査（実施済み）

`03 設計・開発/01 モデル変換/export_torchscript_poc.py`で、既存モデル（`00 runtime/server/models/best_model.pth`、MobileNetV2＋カスタム分類ヘッド、29クラス）をAndroid実機で動かせる形式にエクスポートできるか検証済み（モデルファイル自体も本フォルダの`models/`にコピー済み——複製環境の原則通り、サーバー側の元ファイルは変更していない）。

**分かったこと**：
1. `torch.jit.trace`によるTorchScript化は成功し、元のPyTorchモデルと**完全に数値一致**（誤差0.0）。
2. PyTorchの「モバイル最適化」API（`torch.utils.mobile_optimizer.optimize_for_mobile`）は、このモデルに対して**数値的に大きく異なる出力を生成する不具合を確認**（出力範囲[-41,28]に対し平均絶対誤差23.5）。単純な浮動小数点誤差では説明できない大きさのため、**使用しない**判断とした（最適化なしのtraceのみで保存）。
3. PyTorch公式は「Lite Interpreter」（このtrace済みモデルを実機で動かす従来方式、`org.pytorch:pytorch_android_lite`）自体を**非推奨と案として表示**しており、後継の**ExecuTorch**への移行を推奨している（実行時にDeprecationWarningが出ることを確認）。
4. 最終的に、最適化なしのtrace済みモデル（`.ptl`、10.32MB）を`03 設計・開発/01 モデル変換/exported/board_classifier.ptl`に出力済み。これは動作する（数値的に正しい）が、PyTorch公式の推奨ランタイムではない。
5. **実際の保存済み写真（`runtime/photos/calib_20260621_135539_120137.jpg`、正確な4隅座標は使わずリサイズのみ）でも追加検証済み**（`verify_real_photo.py`）：保存済み`.ptl`を一度ディスクから読み込み直し（exportとは別プロセス）、元モデルと比較——**出力が完全一致（誤差0.0）、PC上の推論時間も同程度（元1.602秒・.ptl 1.601秒）**。合成入力（randn）だけでなく実画像でも数値的な正しさを確認できた。なお`.ptl`の保存・読み込みいずれも、ネイティブ側の`fopen`がこのNAS共有のUNC＋日本語パスを直接扱えず`errno 42`で失敗する問題があり（exportで遭遇した問題と同根、CLAUDE.mdの`cv2.imread`/`imwrite`注意と同じ系統）、ローカル一時ディレクトリ経由でコピーする回避策が必要だった（両スクリプトに実装済み）。

**結論・推奨**：今回出力した`.ptl`はAndroid実機での疎通確認・初期ベンチマークには使える（`org.pytorch:pytorch_android_lite`で読み込み可能なはず、次のタスクで確認）。ただし本格的に進める場合は、非推奨と分かっている方式に乗るより**ExecuTorch（`torch.export`→`to_edge`→`to_executorch`、Android側は`org.pytorch.executorch:executorch-android`）への移行を検討すべき**。

**ExecuTorchの実機検証は、このPC環境特有の問題でブロックされている（要・ユーザー対応）**：`pip install executorch`を実行したところ、`OSError: No such file or directory`で失敗——原因はパッケージ内の一部ファイルパスがWindowsの260文字パス長制限を超えていること（pip自身が表示するヒント通り、「Windows Long Path support」が有効になっていない）。実際に`import executorch`してみると（おそらく失敗したファイル以外は先にコピーされていたため）見かけ上importはできるが、`pip show executorch`は「見つからない」と返す不完全な状態——`.dist-info`が書き込まれておらず、正式にインストールされたとは言えない半端な状態のため、**この状態のまま検証を進めるのは避けた**。

対応案（いずれかが必要、要確認）：
1. Windowsの「長いパスを有効にする」設定をオンにする（レジストリ変更が必要、管理者権限が必要——Claude側からは実行していない。ユーザーの判断・操作が必要な変更のため）。設定後、`pip install executorch`を再実行すれば解決する可能性が高い。
2. WSL（Windows Subsystem for Linux）等、Linux環境でモデル変換（PCでの作業のみ）を行う。Androidアプリ自体のビルドはこれまで通りWindows側（Android Studio）で行い、変換だけ別環境に切り出す。
3. （簡易代替）ひとまず上記の`.ptl`（PyTorch Mobile Lite、非推奨だが数値的に正しいことを確認済み）でAndroid側の実装・比較検証を先に進め、ExecuTorchへの移行は環境問題が解決してから着手する。

**要確認**：どの対応案で進めるか。現時点ではユーザーの環境操作（1）か別環境の用意（2）が必要なため、Claude側だけでは解決できない。3を採用すれば今すぐAndroid側の実装に進めるが、いずれ非推奨APIからの移行作業が発生する。

**要確認**：①ひとまず簡易な`.ptl`（PyTorch Mobile Lite、非推奨だが数値的に正しい）でレスポンス比較の第一弾を行い、良ければExecuTorchへ本格移行する2段階方式で良いか、②最初からExecuTorchで進めるか。前者の方が早く比較結果が出せるが、いずれ非推奨APIからの移行作業が要る。後者は最初の一歩が重いが、やり直しがない。**現時点ではユーザー確認待ちのため、両方の選択肢を残せるよう①の`.ptl`を先に作った（後者を選んでも今回の作業が無駄にはならない——透視変換・クロップ等のAndroid側コードはランタイムが変わっても再利用できる）。**

**追加で見つかったリスク（要・実機確認、Claude側では検証不能）**：`.ptl`をエクスポートしたPC環境のPyTorchは`2.12.0`（かなり新しい）。一方、Android側で読み込む`org.pytorch:pytorch_android_lite`ライブラリはMaven Central上の最新版が`2.1.0`（2023年10月公開、約2年古い——PyTorch MobileがExecuTorchへの移行推奨で更新を止めているため）。エクスポート時とランタイム側のバージョン差が大きく、Lite Interpreterのバイトコード形式が変わっている可能性がある（実際、PCの`torch._C._get_model_bytecode_version()`で確認しようとしたところ「zipアーカイブとして読めない」エラーになった——`torch.jit.load`では問題なく読めるため、ファイル自体は壊れていないが、古いバイトコード検査ツールが今のフラットバッファ形式を認識できていないと見られる）。**これが実際にAndroid実機の`pytorch_android_lite:2.1.0`で読み込めるかどうかは、Android Studio・実機がないこの環境からは確認できない。** 実機でモデル読み込みに失敗した場合は、別途PyTorch 2.1系の環境を用意して`.ptl`を再エクスポートする対応が必要になる可能性がある。

## 4. 画像の前処理（透視変換・クロップ）をどう実装するか（要確認）

サーバー側`recognition.py`の`warp_board`（`cv2.warpPerspective`）と、`predict_board`内のクロップ・リサイズ・正規化（`Resize(224,224)`→`ToTensor`→`Normalize(ImageNet平均/分散)`）を、Android側で再現する必要がある。

**推奨：OpenCV for Android（`org.opencv:opencv-android`）を使い、既存Python実装とほぼ1:1で対応させる。**
理由：透視変換行列の計算・適用はサーバー側と全く同じ数式（`getPerspectiveTransform`/`warpPerspective`）を使うべきで、Androidの標準APIだけで真の4点透視変換を再現するのは煩雑（`android.graphics.Matrix`は4点同次変換を直接サポートしない）。OpenCVは関数名までほぼ同じため移植ミスのリスクが低い。

リサイズ・正規化は、ExecuTorch/PyTorch Mobileどちらのランタイムでも、モデルへの入力テンソルを作る前処理としてKotlin側で実装する必要がある（`Bitmap`→`FloatArray`、ImageNet正規化の平均・分散値はサーバー側`_transform`と完全に同じ値を使う——ここがずれると認識精度が劣化するので要注意）。

**要確認→実装済み（2026-06-23）**：OpenCV for Android採用、`org.opencv:opencv:5.0.0`（Maven Central、4.9.0以降は公式が直接配布、OpenCV Managerアプリ不要）を`03 設計・開発/02 Androidアプリ開発/gradle/libs.versions.toml`に追加済み。Gradle Sync・実機での動作確認はまだ（Android Studioが必要、Claude側では検証不能）。

## 5. アプリの複製方法（要確認→採用・実装済み）

「新しく複製した環境で新しく開発する」という前提に対し、以下のいずれかで対応：

- **(採用) 既存Androidプロジェクト（`03 設計・開発/01 Androidアプリ開発/`相当）をフォルダ単位でコピーし、本PJの`03 設計・開発/02 Androidアプリ開発/`に置いた。パッケージ名を`com.example.shogiban_kaiseki_appli_ondevice`に変更し、別アプリとして実機に共存インストールできる。**
- (代替・今回は不採用) 最初から最小限の単一画面アプリを新規作成。

**実際に実装した範囲（当初の推奨案から、検証目的に合わせて以下に絞った）**：
カメラ撮影・4隅タップUI・対局フロー（キャリブレーション/対局中/棋譜一覧の各画面）はこのPoCでは複製していない。理由：①これらの画面自体は本検証（CNN処理オンデバイス化の速度比較）に関係しない、②Claude側にはAndroid実機もカメラもなく、ライブのタップ操作・撮影フローを検証する手段がない。代わりに、**PCで自動検出済みの実写真2枚（4隅座標つき、いずれも初期配置と完全一致を確認済み）をAndroidの`assets/`にバンドルし、その写真に対して「オンデバイス推論」と「サーバー推論」を同条件で実行して時間を比較するだけの単一画面アプリ**にした。撮影・タップ操作のUXコストを排除した、CNN処理そのものの速度差を測るための専用ベンチマークアプリという位置づけ。

実装場所：`03 設計・開発/02 Androidアプリ開発/`
- `app/src/main/java/.../recognition/OnDeviceRecognition.kt`：`recognition.py`の`warp_board`・`predict_board`をOpenCV+PyTorch Mobile Liteで再現。
- `app/src/main/java/.../recognition/TestPhotos.kt`：バンドル写真2枚の4隅座標と、初期配置の正解ラベルグリッド（比較用）。
- `app/src/main/java/.../MainActivity.kt`：写真選択→「オンデバイスで実行」/「サーバーで実行（比較用）」の2ボタン、処理時間と正解一致を表示。
- `app/src/main/assets/`：`test_photo_1.jpg`・`test_photo_2.jpg`（実写真）・`board_classifier.ptl`（変換済みモデル）。

**サーバー比較について重要な注意**：ベンチマークの「サーバーで実行」は、本番の8000番ポートではなく**別ポート（8001番）で起動した同じ既存サーバーコード**を呼ぶ設計にした（`ApiConfig.BASE_URL`参照）。理由：本番サーバーは対局セッション状態をシングルトンで持っており（`session.py`）、ベンチマークの繰り返し呼び出しでその状態を壊すリスクがあるため。起動方法：`00 runtime/server`で`uvicorn main:app --host 0.0.0.0 --port 8001`（既存コードは無変更、同じコードをもう一つ起動するだけ）。

**Gradle Sync・実機ビルド確認はまだ（要・ユーザー対応）**：このCLI環境にはAndroid StudioもAndroid実機もないため、Gradle Syncが通るか、実機でアプリが起動するか、`.ptl`が`pytorch_android_lite:2.1.0`で実際に読み込めるか（上記3節のバージョン差リスク）は未確認。Android Studioで開いてGradle Syncし、実機（または既存アプリと同じ機種）にインストールして「オンデバイスで実行」ボタンを押してもらう必要がある。

## 6. 比較方法（検証手順案）

1. **オフライン比較（実機不要、先行して可能）**：サーバーの`runtime/photos/`に保存済みの実際のキャリブレーション写真を使い、(a)既存サーバー側`predict_board`の処理時間、(b)オンデバイス版モデル（`.ptl`）をPC上のPythonから読み込んで同じ画像で推論した処理時間、を比較する。ネットワーク往復を含まない「CNN処理そのものの速度差」をまず確認する。
2. **オンデバイス実機ベンチマーク**：複製したAndroidアプリで、実際の盤を撮影→オンデバイス推論→結果表示までの時間を計測（`System.currentTimeMillis()`等で区間計測、既存の`NetworkTiming.kt`の手法を参考に）。
3. **既存（サーバー送信）との比較**：同じ撮影シーン・同じWiFi環境で、既存アプリ（サーバー送信版）の「撮影→結果表示」までの時間と比較する。
4. **認識精度の確認**：オンデバイス版とサーバー版で、同じ写真に対する9×9認識結果が完全一致するか（量子化等を行わない限り基本的に一致するはずだが、念のため確認）。

**移行判断基準（案、要確認）**：オンデバイス版が既存（サーバー送信）版より明確に速く（例：体感できる差、具体的な閾値は要相談）、かつ認識精度が劣化していないこと。
