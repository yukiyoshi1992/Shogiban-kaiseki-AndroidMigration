# アーキテクチャ検討（ネイティブアプリ化）

**確定事項（2026-06-20 ユーザー確認済み）**
- 前PJのモデル・盤面解析ロジックは本リポジトリに複製して取り込む（2節の通り）。
- Androidアプリの技術選定（Kotlin + Jetpack Compose + CameraX + Retrofit）は提案通り採用。開発はAndroid Studioで行う（IDEとしてのAndroid Studio／言語としてのKotlin／ライブラリとしてのCompose・CameraX・Retrofitは併存するもので矛盾しない）。開発用の作業ディレクトリは `03 設計・開発/01 Androidアプリ開発/` 配下を想定し、ユーザーがAndroid Studioでプロジェクト作成後、そこにclaude codeを招待してコード実装を進める。

`ネイティブアプリ化_要件.xlsx`のTOBEアーキテクチャ（スマホアプリ⇄FastAPI⇄前回作成の盤面解析プログラム⇄NAS）を具体化したもの。「今後の検討事項」シートNo.4（解析プログラム連携の具体化）に対応する。

## 1. 全体構成

```
[Androidアプリ]
   │ HTTP/JSON（写真はmultipart）, LAN内のみ
   ▼
[FastAPIサーバ]  ※前回プロジェクトと同じWindows PC上で稼働
   │ 盤面認識・指し手判定ロジックを内包（前PJから移植）
   │ ファイルI/O
   ▼
[Buffalo NAS]  ※前PJと共用のストレージ（KIF保存先）
```

- 同時に1局のみ。サーバプロセス内にシングルトンで「現在の対局状態」を保持する（リクエストごとに状態を渡さない）。
- 外部公開なし。LAN内専用。

## 2. 前PJ資産の取り込み方針（検討事項④の核心）

前PJ `Shogiban-kaiseki-tool` は完了・読み取り専用アーカイブとして凍結されている。本プロジェクトでは、前PJのコード・モデルを**本リポジトリに移植（コピー＆必要な改修）**して取り込む方針を提案する。

取り込む対象（前PJ `src/run_realtime.py` および `src/models/`）：
- `best_model.pth` + `model_meta.json`（CNNモデル本体）
- `load_model()` / `predict_board()`（盤面認識）
- `order_points()` / `compute_calib()`（4隅タップ→透視変換行列。`detect_red_circles()`等の自動検出部分は不要）
- `classify_frame()`（指し手判定の中核ロジック）
- `save_kif()` 周辺のKIF生成ロジック

移植時に修正すること：
- `classify_frame`の採用判定（`MOVE_DIFF_THRESHOLD <= 2`）を、**diffが完全に0になる候補のみ採用、それ以外はdepth2へ、最終的にダメなら`"error"`**に変更する（CLAUDE.mdに記載済みの既知の弱点。本プロジェクトはBluetoothボタン1クリック1枚撮影のため、二重発火・撮り忘れで誤った手を静かに採用するリスクが前PJより高い）。
- KIF出力のUSI→KIF表記変換バグ（移動元の半角数字化）は前PJで修正済みのものを引き継ぐ。

**要確認**：前PJのコードをそのまま複製する方針でよいか。代替案として「前PJリポジトリを編集せず参照し続ける」も技術的には可能だが、前PJは凍結・編集禁止の方針のため、本プロジェクト側での改修（上記の閾値修正など）ができなくなる。複製を推奨。

## 3. APIサーバ（FastAPI）設計

### セッション状態（サーバ内メモリで保持）
- `python-shogi`の`Board`オブジェクト（現局面の真実）
- キャリブレーション行列（4隅タップ座標から計算した透視変換）
- 進行中KIFファイルのハンドル
- 状態：`idle → calibrating（撮影・12点指定中） → ready（人間確認待ち） → playing（対局中） → finishing（終局処理中） → idle`
  （前PJのStreamlit版 `idle→ready→playing→(finishing)→idle` を踏襲）

### エンドポイント（案）
| メソッド | パス | 用途 |
|---|---|---|
| POST | /calibration/photo | 初期配置の撮影画像＋12点座標を送信→9×9認識結果を返す（人間確認用） |
| POST | /calibration/confirm | 確認／補正後の盤面を確定し対局開始（state→playing） |
| POST | /move | 1手分の撮影画像を送信→`classify_frame`で指し手判定→成功時はKIF追記＋読み上げテキストを返す。失敗時はリトライ指示または`error` |
| POST | /game/end | 終局通知、KIF確定保存（state→idle） |
| GET | /games?date=... | KIF一覧（対局開始日時で絞り込み） |
| GET | /games/{id} | 指定KIFの内容取得（共有・KENTO連携用） |

## 4. Androidアプリ内部構成（推奨案）

- **言語**：Kotlin（現行のAndroidネイティブ開発の標準。Javaを選ぶ特別な理由はないため推奨）
- **UI**：Jetpack Compose（宣言的UI。新規プロジェクトでの標準的選択）
- **通信**：Retrofit + OkHttp（画像は`multipart/form-data`）
- **カメラ**：CameraX（Android標準のカメラAPI。プレビュー常時表示・撮影に対応）
- **画面構成**：
  1. キャリブレーション画面：カメラ起動→12点タップ→9×9グリッドオーバーレイで確認→補正→確定
  2. 対局画面：カメラプレビュー常時表示、Bluetoothシャッター（音量キー）押下検知、結果表示、TTS再生
  3. KIF一覧・共有画面：日付フィルタ、共有シート（LINE/メール）、KENTO連携
- **非機能**：
  - WakeLock（`PowerManager.PARTIAL_WAKE_LOCK`または`FLAG_KEEP_SCREEN_ON`）で対局中のスリープ防止（検討事項⑤の一部解消。実装詳細は設計フェーズで）
  - Bluetoothシャッター＝音量キーの`KeyEvent`をActivityでフックして撮影トリガーにする

## 6. 本日のギャップリストへのユーザー回答（2026-06-20）

xlsxへの追記ではなく、軽量にこのファイルへ記録（理由：xlsxをスクリプトで直接書き換えると埋め込み図・アイコンが壊れるリスクがあるため。本体への反映を希望する場合はユーザーに別途確認）。

- **CNNモデルの再学習要否（旧ギャップNo.12）→ 解消（再学習不要）**：スマホ本体（カメラハードウェア）は変わらず、カメラアプリ（前PJの汎用カメラアプリ→本プロジェクトのネイティブアプリ内カメラ）だけが変わるため、画角・画質は前PJ学習時と同条件のはず。再学習は不要という前提で進める。
- **認識リトライ失敗後の途中補正・対局再開（旧ギャップNo.6）→ 追加要望として記録、対応は後回し**：前PJは「1回失敗したら諦める」仕組みだったが、本プロジェクトはリアルタイム処理のため、エラーが起きた時点で人間が気づいて途中補正できる可能性がある。ただしこれは今回のタスク（ネイティブアプリ化）に紐づかない追加要望のため、要件としては記録するが実装の優先度は低く、対応は後回し。
- **キャリブレーション手動補正の具体的な操作方法（旧ギャップNo.7）→ 解消**：個別マスの補正は行わない。間違っているマスは1つ・2つではなく多数になることがほとんど（≒四隅タップのズレが原因であることが多い）ため、補正方法は**12点タップキャリブレーションのやり直し**のみでよい（前PJのStreamlit版の運用と同じ）。9×9グリッドオーバーレイは「ズレているかどうかを人間が判断する」ための確認表示であり、個別セル編集UIは不要。
